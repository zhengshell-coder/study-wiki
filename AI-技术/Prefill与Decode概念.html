Prefill 与 Decode

LLM 推理两阶段 · 系统性讲解

    第 0 章 · 导读

    为什么 LLM 推理要分成两个阶段？

    同一个模型、同样的权重，推理时却表现出两副完全不同的面孔：处理你的输入时它像一台算力机器，吐出回答时它又像一台搬运内存的机器。理解这两副面孔——Prefill 与 Decode——是看懂现代推理系统一切优化的钥匙。

    一个直观的现象

    你给 ChatGPT 这样的产品发一段长长的提问，会注意到一个规律：无论提问多长，第一个字蹦出来之前总有一段等待；而一旦开始输出，文字便以相对稳定的速度一个接一个地流出来。

    这不是巧合，而是底层两个阶段的直接体现：

    • Prefill 预填充：模型一次性"读完"你的整段输入（prompt），为生成做准备。等待第一个字的时间，大头花在这里。
    • Decode 解码：模型逐个 token 地生成回答，一次只产出一个 token，循环往复直到结束。你看到的"流式输出"就是它。

    为什么这件事值得专门讲一章

    如果两个阶段只是"先读后写"这么简单，那确实不必大费周章。关键在于：它们的硬件瓶颈完全相反。

    • Prefill 把成百上千个 token 一起喂进 GPU，矩阵又大又满，卡在算力（compute-bound）——GPU 的计算单元被压榨到极限。
    • Decode 每步只处理 1 个新 token，计算量极小，却要把整个模型权重和不断增长的缓存反复从显存里搬出来，卡在访存带宽（memory-bound）——GPU 算力大量闲置，在等数据。

    同一块 GPU，在 Prefill 时利用率可以接近 100%，在 Decode 时算力利用率常常只有个位数百分比。几乎所有现代推理优化——KV Cache、continuous batching、chunked prefill、PD 分离、投机解码——本质上都是在调和这对矛盾。

    本手册的主线我们会先补齐自回归生成的基础（第 1 章），再分别拆解 Prefill（第 2 章）与 Decode（第 3 章），用 KV Cache（第 4 章）把两者串起来；然后从计算特性（第 5 章）和性能指标（第 6 章）建立"度量"的视角，最后落到系统优化（第 7 章）与真实框架（第 8 章）。第 9 章是一页速查表。

    第 1 章 · 基础

    自回归生成：两阶段从何而来

    Prefill 和 Decode 不是谁设计出来的"功能"，而是自回归（autoregressive）生成方式 + 注意力机制这两个事实自然导出的结果。先把这层基础铺平，后面一切都顺理成章。

    1.1 语言模型在做什么：预测下一个 token

    一个 GPT 类模型，本质上是一个函数：给定到目前为止的 token 序列，输出"下一个 token 是什么"的概率分布。

    P( xt | x1, x2, …, xt-1 )已知前面所有 token，预测第 t 个 token 的概率分布

    注意两个关键词：token（不是字也不是词，而是分词器切出的子词单元，比如 "推理" 可能是一个 token，"inference" 可能被切成 "infer"+"ence"）；以及条件——预测第 t 个时，模型能看到前面全部 t−1 个 token。

    1.2 "自回归"：把上一步的输出当作下一步的输入

    生成一段文本，模型并不是一次性写完，而是一个字一个字地"接龙"：

    1. 拿到当前序列，预测下一个 token 的分布；
    2. 按采样策略（贪心 / top-p / 温度等）选出一个 token；
    3. 把这个新 token 追加到序列末尾；
    4. 回到第 1 步，直到生成结束符或达到长度上限。

    "用自己刚生成的输出作为下一步输入"——这就是自回归。它有一个无法回避的特性：串行。第 t 个 token 必须等第 t−1 个生成完才能开始，因为它要把 xt-1 当输入。这正是 Decode 阶段慢、难以并行的根本原因。

    那 Prefill 为什么能并行？因为输入的 prompt 是已知的、完整的。整段 prompt 的所有 token 同时摆在面前，不存在"等上一个算完"的依赖，于是可以一次性塞进 GPU 并行处理。差别的源头就在这里：prompt 是已知的（可并行），续写是未知的（必须串行）。

    1.3 注意力机制：为什么要回看前面所有 token

    Transformer 的核心是自注意力（self-attention）。每生成一个位置的表示，模型会让当前 token 去"查询"前面每一个 token，按相关性加权汇总它们的信息。具体地，每个 token 会被投影成三个向量：

    • Q（Query，查询）：我想找什么；
    • K（Key，键）：我能被什么样的查询匹配到；
    • V（Value，值）：如果被匹配到，我贡献什么信息。

    当前 token 的 Q 与前面所有 token 的 K 做点积得到注意力权重，再用这些权重对所有 V 加权求和。一句话：

    Attention(Q, K, V) = softmax( Q·KT / √d ) · V当前 token 的 Q，对照历史所有 token 的 K 与 V

    1.4 关键洞察：历史 token 的 K、V 是"可复用"的

    这是连接两个阶段、并通往 KV Cache 的最重要一步。请仔细看：

    当模型生成第 t 个 token 时，它需要前面 1…t−1 所有 token 的 K 和 V。当它继续生成第 t+1 个 token 时，又需要 1…t 所有 token 的 K 和 V——其中 1…t−1 这部分和上一步完全一样，因为这些 token 已经定下来了，它们的 K、V 不会改变。

    于是有了一个巨大的省力空间既然历史 token 的 K、V 算一次就永远不变，为什么每步都重算？把它们缓存起来，每生成一个新 token，只算这一个新 token 的 K、V 并追加进缓存即可。这个缓存就是 KV Cache（第 4 章详解），它是 Decode 阶段高效运转的前提，也是显存占用的大头。

    1.5 把两阶段重新定义一遍

    有了上面的铺垫，现在可以给出更准确的定义：

    • Prefill 对已知的整段 prompt做一次前向传播，并行计算出所有 prompt token 的 K、V 存入缓存，同时得到"预测第一个输出 token"所需的信息。
    • Decode 进入串行循环：每步把上一个 token 喂入，只计算它一个的 Q/K/V，用它的 Q 去注意缓存里的全部历史 K/V，预测下一个 token，再把新 K/V 追加进缓存。

    两个阶段跑的是同一个模型、同一套权重、同样的注意力公式，区别只在于：一次处理多少个 token（Prefill 是一批，Decode 是一个），以及K/V 是新算还是从缓存读。正是这一点差别，导致了下面几章要讲的截然不同的性能行为。

    第 2 章 · Prefill

    Prefill 阶段：一次性吃透整段输入

    Prefill 干一件事：把用户给的整段 prompt 跑一遍前向传播，产出两样东西——全部 prompt token 的 KV Cache，以及预测第一个输出 token 所需的最后一层输出。因为整段输入已知，它可以高度并行，是典型的算力密集型负载。

    2.1 输入与输出

    假设 prompt 有 N 个 token。Prefill 的一次前向传播：

    • 输入：N 个 token 的 embedding，形状大致为 [N, d]（d 是隐藏维度）。
    • 过程：N 个 token 一起穿过模型的每一层（注意力 + FFN），彼此之间通过注意力交换信息。
    • 输出：① 每一层、每个 token 的 K 和 V，全部写入 KV Cache；② 最后一个位置的输出向量，经过输出层得到"第一个生成 token"的概率分布。
    注意一个常被忽略的点Prefill 计算了 N 个位置的全部表示，但真正用来预测下一个 token 的只有最后一个位置。前面 N−1 个位置的输出 logits 在普通生成中并不需要——它们存在的意义是把 K、V 算出来留给后续 Decode 用。所以 Prefill 的产物里，KV Cache 才是给未来用的主角。

    2.2 为什么 Prefill 可以并行

    因为 prompt 的所有 token 此刻全部已知。注意力虽然是"因果"的（第 t 个只能看 1…t），但这只是一个掩码（causal mask），并不妨碍把 N 个 token 当成一个矩阵一起算——GPU 把它变成一次大的矩阵乘法 [N, d] × [d, d]，N 越大，矩阵越"胖"，GPU 的并行计算单元就越满。

    2.3 计算量从哪来：两块大头

    Prefill 的浮点运算（FLOPs）主要来自两部分：

    • 线性投影 / FFN：与 token 数 N 成正比，是计算量的主体。粗略地，每个 token 的前向计算约为 2 × 参数量 次浮点运算，整段 prompt 就是 ≈ 2 × N × 参数量。
    • 注意力本身：Q·KT 和对 V 加权，复杂度随 N² 增长。短 prompt 时占比不大，但 prompt 很长时会变得显著——这也是超长上下文成本高的原因之一。

    2.4 Prefill 的性能特征

    维度Prefill 的表现
    一次处理的 token 数很多（整段 prompt，N 个）
    并行度高，N 个 token 同时算
    瓶颈算力受限（compute-bound）
    GPU 利用率高，容易接近峰值算力
    对延迟的影响决定 首字延迟 TTFT（见第 6 章）
    耗时如何随输入增长大致随 prompt 长度线性增长（注意力部分更快）
    一个实践后果因为 Prefill 的耗时随 prompt 长度增长，超长 prompt 会显著拉高首字延迟。一段 8K token 的上下文，光 Prefill 就可能占掉用户感知到的等待的绝大部分。这也是为什么"减少不必要的上下文""prompt 缓存（prefix caching）"等手段对体验影响很大——它们直接砍掉 Prefill 的工作量。

    2.5 小结

    Prefill 是"读题"阶段：把已知的整段输入并行跑一遍，攒出 KV Cache，并迈出生成的第一步。它算得多、并行好、吃算力，决定了用户等待第一个字的时间。下一章我们看它的反面——Decode。

    第 3 章 · Decode

    Decode 阶段：逐字作答的串行循环

    Decode 是生成回答的阶段。它每一步只做一件小事：拿上一个 token，算出下一个 token。计算量小得可怜，却因为必须串行、且每步都要把整个模型权重和缓存从显存搬一遍，成了典型的访存密集型负载，也是推理慢、贵的主因。

    3.1 一步 Decode 做了什么

    假设 Prefill 已经跑完，KV Cache 里已有 prompt 的全部 K、V。现在进入循环，每一步：

    1. 取上一个生成的 token（第一步取的是 Prefill 预测出的首个 token），形状 [1, d]——注意只有 1 个 token。
    2. 让它穿过每一层。在注意力里，只为这 1 个新 token 计算 Q、K、V。
    3. 用它的 Q，去注意 KV Cache 里的全部历史 K、V（prompt + 之前生成的所有 token）。
    4. 把这 1 个新 token 的 K、V 追加进 KV Cache。
    5. 最后一层输出 → logits → 采样出下一个 token。
    6. 回到第 1 步。直到生成结束符或达到上限。

    3.2 为什么 Decode 必须串行

    第 t+1 个 token 的输入，是第 t 个 token——而第 t 个 token 要等这一步算完、采样出来才知道。未来的输入依赖当前的输出，所以无法像 Prefill 那样把多个 token 摊开一起算。这是自回归的本质约束（第 1 章），不是工程没做好。

    那 Prefill 不也是因果的吗，怎么能并行？因果掩码只限制"谁能看谁"，不限制"谁先算"。Prefill 的 token 全部已知，可以同时摆上桌；Decode 的 token 一个还没生出来，逼着你排队。已知 vs 未知，是可并行与否的分水岭。

    3.3 致命的不对称：算得少，搬得多

    Decode 的每一步，计算量只相当于处理 1 个 token，少到 GPU 的计算单元几乎是闲着的。但要完成这一步，GPU 必须：

    • 把整个模型的权重（几 GB 到上百 GB）从显存读进计算单元——哪怕只算 1 个 token，权重也得全读一遍；
    • 把整个 KV Cache（随生成长度不断变大）读出来做注意力。

    于是出现了一个尴尬的局面：搬运数据的时间远远大于计算的时间，GPU 大部分时候在等显存。这就是访存受限（memory-bound）。一个直观的比方：让一位大厨（强大的算力）只为切一根葱，却要求他每次都把整个仓库的食材（权重 + 缓存）搬到台前再搬回去——切葱的功夫可以忽略，全耗在搬运上。

    这把矛盾点了出来，也指明了出路既然每步搬一次权重只切一根葱太浪费，那就一次多切几根——把多个请求的"当前 token"凑成一批，共享同一次权重读取。这就是 batching（第 7 章），是提升 Decode 吞吐最有效的手段，原因正是它摊薄了昂贵的权重访存。

    3.4 Decode 的性能特征

    维度Decode 的表现
    一次处理的 token 数1 个（每个请求）
    并行度单请求内为零（串行）；跨请求可 batch
    瓶颈访存受限（memory-bound）
    GPU 算力利用率低，常只有个位数百分比
    对延迟的影响决定 每字延迟 TPOT 与总生成时间（见第 6 章）
    随生成变长的代价KV Cache 越来越大，每步访存越来越多，且持续占显存

    第 4 章 · 核心机制

    KV Cache：把两个阶段缝在一起

    KV Cache 是理解推理系统的枢纽。它由 Prefill 创建，被 Decode 反复读取和追加，是"省掉重复计算"的功臣，也是"吃光显存、限制并发"的元凶。搞懂它，前后几章全部连成一条线。

    4.1 它缓存的到底是什么

    回顾第 1 章：注意力里，每个 token 都会产出 Q、K、V 三个向量。其中历史 token 的 K 和 V 一旦算出就不再改变（因为这些 token 定下来了）。KV Cache 缓存的正是每一层、每个历史 token 的 K 向量和 V 向量。注意：

    • 缓存的是 K 和 V，不缓存 Q——Q 是"当前 token 想查什么"，每步都是新的，没有复用价值。
    • 缓存是逐层的：模型有多少层，就有多少份 K/V 缓存。
    • 它随生成只增不减：每生成一个 token，就往每层追加一份新的 K、V。

    4.2 没有 KV Cache 会怎样

    设想关掉缓存：生成第 t 个 token 时，要重新计算 1…t−1 所有 token 的 K、V；生成第 t+1 个时，又把 1…t 全重算一遍……每一步都把历史从头算起。总计算量会从"随长度线性"恶化成"随长度平方"，长文本生成将慢到不可用。

    KV Cache 的本质它是一个用显存换计算的经典权衡：花一块持续增长的显存，把"每步重算全部历史"省成"每步只算 1 个新 token + 读一遍缓存"。这是自回归生成能高效跑起来的前提。

    4.3 两阶段视角下的 KV Cache 生命周期

    4.4 算一笔显存账（理解量级即可）

    KV Cache 的大小有一个直观的公式，不必背，但要理解它和哪些量成正比：

    KV 显存 ≈ 2 × 层数 × KV头数 × 头维度 × 序列长度 × 批大小 × 精度字节2 代表 K 和 V 各一份

    关键结论：

    • 与序列长度成正比——上下文越长、生成越长，缓存越大。这是长上下文昂贵的核心原因之一。
    • 与并发请求数（批大小）成正比——同时服务的用户越多，缓存总量越大。
    • 它和模型权重分开占显存。权重是固定开销，KV Cache 是随负载浮动的开销。
    它常常才是并发的真正天花板很多人以为"能服务多少并发"由算力决定，实际上常常是 KV Cache 把显存吃满了，导致无法再接新请求。所以省显存的 KV 优化（下一条）几乎等价于"提升并发能力"。

    4.5 围绕 KV Cache 的优化（先建立框架认知）

    这些会在第 7 章展开，这里先知道它们都在解决"KV Cache 太大 / 太碎 / 不复用"的问题：

    • GQA / MQA（分组 / 多查询注意力）：让多个注意力头共享同一份 K、V，直接把 KV 缓存缩小数倍。现代大模型几乎都用，是省显存的"出厂设置"。
    • PagedAttention（分页）：像操作系统管理内存那样，把 KV Cache 切成小块按需分配，消除碎片浪费——vLLM 的招牌设计。
    • Prefix Caching（前缀缓存）：多个请求若共享相同前缀（如同一段 system prompt），它们的 KV 可以复用，省掉重复 Prefill。
    • 量化 / 卸载：把 KV 用更低精度存储，或暂时挪到 CPU 内存，腾出显存。

    4.6 小结

    KV Cache 是"用显存换计算"的核心机制：Prefill 创建它，Decode 复用并扩展它。它让自回归生成变得可行，同时把瓶颈从"重复计算"转移到了"访存与显存容量"——这恰好解释了为什么 Decode 是 memory-bound，也为什么显存常是并发的上限。带着这个认识，下一章我们正式建立"算力受限 vs 访存受限"的分析框架。

    第 5 章 · 计算特性

    算力受限 vs 访存受限：一张图说清

    前几章反复提到 Prefill 是 compute-bound、Decode 是 memory-bound。这一章把这句话讲到底——用一个叫"算术强度"的概念和 roofline 模型，让你看清两个阶段为什么必然落在硬件性能曲线的两端。

    5.1 GPU 有两种"速度上限"

    任何一段计算在 GPU 上跑，都同时受两个能力约束：

    • 算力（compute）：每秒能做多少次浮点运算，单位 FLOP/s。比如某 GPU 峰值约几百 TFLOP/s。
    • 访存带宽（memory bandwidth）：每秒能从显存（HBM）搬多少字节进计算单元，单位 GB/s 或 TB/s。

    一段计算最终跑多快，取决于这两者中先到达上限的那个。如果计算单元先满，就是算力受限；如果带宽先满（计算单元在等数据），就是访存受限。

    5.2 算术强度：判断卡在哪一边的标尺

    算术强度（arithmetic intensity） = 这段计算的浮点运算次数 ÷ 它需要搬运的字节数。

    算术强度 = FLOPs / Bytes每搬运 1 字节数据，能"喂饱"多少次计算
    • 算术强度高：搬一点数据能算很久 → 计算单元忙得过来 → 多半是算力受限。
    • 算术强度低：搬一堆数据只算一点点 → 计算单元在等数据 → 多半是访存受限。

    5.3 把两个阶段放进 Roofline 图

    这张图是整套理论的浓缩。横轴是算术强度，纵轴是实际能达到的性能。斜线代表"被带宽卡住"时的上限，平线代表"被算力卡住"时的上限，两者交点是拐点（ridge point）。

    • Prefill 一次算 N 个 token，复用同一批权重做大量计算，算术强度高，落在拐点右侧、贴着算力屋顶。增大算力才能更快。
    • Decode 一次只算 1 个 token，却要把整套权重读一遍，算术强度极低，落在拐点左侧斜坡上。此时再强的算力也没用，只有更高带宽或更高的有效强度才有帮助。

    5.4 为什么 batching 能救 Decode，却对 Prefill 帮助有限

    这是 roofline 框架最有用的推论。Batching = 把 B 个请求的"当前 token"凑成一批一起算。

    • 对 Decode：权重只需读一次，却服务了 B 个 token 的计算 → 计算量涨了 B 倍，访存几乎没变 → 算术强度提高 B 倍，在 roofline 上沿斜坡向右移动，逼近拐点。吞吐随之大涨。这就是为什么所有推理系统都拼命攒 batch。
    • 对 Prefill：它本来就在算力屋顶下，算术强度已经很高，再加 batch 只是让计算更多、但单位性能已经触顶 → 主要是排队/吞吐层面的收益，而非"解锁闲置算力"。
    一句话记住Prefill 缺的是"算力时间"，Decode 缺的是"把昂贵的权重读取摊给更多 token"。前者靠更强的卡 / 更多并行，后者靠 batching、GQA、量化等提高有效算术强度的手段。

    5.5 这把后面所有优化都解释了

    优化手段本质上在做什么
    Continuous batching持续把更多请求拼进 Decode 批次，抬高算术强度
    GQA / MQA缩小 KV，减少 Decode 每步的访存字节
    KV 量化同样减少访存字节
    Chunked prefill把 Prefill 切块，与 Decode 混跑，填满 GPU 空隙
    投机解码一次验证多个候选 token，单次权重读取产出更多 token

    看懂了 roofline，你会发现第 7 章的各种技巧都不是孤立的招式，而是同一个目标的不同侧面：让算力受限的 Prefill 别等，让访存受限的 Decode 别浪费算力。下一章先建立衡量这一切的指标。

    第 6 章 · 性能指标

    怎么衡量：TTFT、TPOT、吞吐与延迟

    两个阶段对应两类用户体验，也对应两组截然不同的指标。分不清它们，就会优化错方向。这一章建立一套能把"快不快""划不划算"说清楚的度量语言。

    6.1 两个核心延迟指标

    TTFT · 首字延迟

    Time To First Token：从请求到达，到吐出第一个 token 的时间。

    主要由 Prefill 决定，随 prompt 长度增长。它决定了"反应快不快"。

    TPOT · 每字延迟

    Time Per Output Token：进入流式输出后，相邻两个 token 之间的平均间隔（也叫 ITL，token 间延迟）。

    由 Decode 决定，体现"打字速度"。常用每秒 token 数表示。

    6.2 一个请求的总延迟怎么拆

    把两个指标拼起来，就得到端到端延迟的直观公式：

    总延迟 ≈ TTFT + TPOT × (输出 token 数 − 1)首字的等待 + 之后每个字的累加
    体验上的差异TTFT 高 = 用户盯着空白等很久才有反应；TPOT 高 = 文字一顿一顿地往外蹦。短输出（如分类、判断）对 TTFT 敏感；长输出（如写文章、生成代码）对 TPOT 敏感，因为它要乘以很多 token。

    6.3 吞吐：另一套视角

    延迟关心"单个请求多快"，吞吐关心"系统单位时间能干多少活"，是成本的核心。

    • Token 吞吐：每秒生成多少 token（tokens/s），通常按整机统计。Decode 的 batching 直接拉高它。
    • 请求吞吐：每秒完成多少请求（req/s 或 QPS）。

    6.4 吞吐与延迟的根本矛盾

    提高吞吐最有效的办法是增大 batch（第 5 章解释过，这能抬高 Decode 的算术强度）。但 batch 越大：

    • 单个请求要和更多请求共享算力、排更长的队，单请求的 TPOT/TTFT 往往变差；
    • KV Cache 占用更多显存，可能触及容量上限。

    于是形成经典权衡：追求低延迟就难有高吞吐，追求高吞吐就要牺牲单请求延迟。

    6.5 实践中常用的服务等级目标（SLO/SLA）

    线上服务通常不看平均值，而看尾延迟（如 P95、P99），并对两个阶段分别设目标，例如："P99 TTFT < 500ms 且 P95 TPOT < 50ms"。然后在满足这组约束的前提下，把吞吐(即把硬件成本)压到最低。

    为什么要分两阶段设指标因为它们由不同阶段、不同瓶颈决定，优化手段也不同。TTFT 不达标 → 往 Prefill 想办法（prefix caching、chunked prefill、减上下文）；TPOT 不达标 → 往 Decode 想办法（更小的 KV、投机解码、给 Decode 更多带宽）。把它们混成一个"延迟"指标，就会失去优化的抓手。

    有了这套度量语言，第 7 章就能讲清各种系统优化"改善了哪个指标、付出了什么代价"。

    第 7 章 · 系统优化

    系统优化：调和两阶段的矛盾

    前面建立的所有概念，在这一章兑现成真实的工程手段。每一种优化都对应一个明确的问题：Prefill 太占场、Decode 太浪费、两者抢资源、KV 太大。理解"它解决什么"，比记住名字重要得多。

    7.1 矛盾的根源：Prefill 与 Decode 抢同一块 GPU

    一台推理服务器要同时面对很多请求：有的刚到、需要 Prefill，有的在输出中、需要 Decode。这两类活儿性格相反——Prefill 吃算力、一来就是一大坨；Decode 吃带宽、零零碎碎但量大。怎么把它们安排在同一块（或同一组）GPU 上而不互相拖累，就是调度的核心难题。下面的手段，基本都在回答这个问题。

    7.2 Continuous Batching（连续批处理）

    问题：传统静态 batching 要等一批请求都到齐才开跑，且必须等批里最慢的请求结束才能换下一批——短请求被长请求拖着，GPU 大量空转。

    做法：以 token 为粒度动态拼批。每一步结束后，已生成完的请求立刻退出、腾出位置，新到的请求随时插入当前正在跑的 Decode 批次。批次像流水一样持续滚动，不再"对齐等待"。

    收益：把 Decode 的算术强度持续维持在高位（第 5 章），GPU 利用率和吞吐大幅提升。这是现代推理引擎（vLLM、TGI 等）的标配，常被认为是吞吐提升最大的单项优化。

    主要改善 Decode 吞吐

    7.3 PagedAttention 与分页 KV Cache

    问题：KV Cache 随生成增长，长度事先不知道。若给每个请求预留一整段连续显存，会产生大量内部碎片（预留多了浪费）和外部碎片（凑不出连续大块），显存利用率很低。

    做法：借鉴操作系统虚拟内存，把 KV Cache 切成固定大小的"块（page）"，按需分配、用逻辑页表映射。请求要多少给多少，不必连续。

    收益：显存碎片几乎消除，同样显存能塞下更多并发请求 → 更大的有效 batch → 更高吞吐。它还让前缀共享变得容易（见下）。这是 vLLM 的招牌设计。

    提升显存利用率 → 提升并发与吞吐

    7.4 Prefix Caching（前缀缓存）

    问题：很多请求开头是一样的——相同的 system prompt、相同的少样本示例、多轮对话里相同的历史。每次都对这段相同前缀重新做 Prefill，是纯重复劳动。

    做法：把已经算过的前缀对应的 KV Cache 留存下来，遇到相同前缀的新请求直接复用，跳过这部分 Prefill。配合分页很自然——共享前缀就是共享那些页。

    收益：命中时 TTFT 大幅下降、Prefill 算力开销骤减。对话类、Agent 类、固定模板类场景收益尤其明显。

    主要改善 TTFT 与 Prefill 成本

    7.5 Chunked Prefill（分块预填充）

    问题：一个超长 prompt 的 Prefill 是一大坨算力密集型任务，它一旦上 GPU，正在进行的 Decode 步骤就被卡住排队，导致其他用户的 TPOT 突然飙高（卡顿）。

    做法：把长 Prefill 切成若干小块，分多步执行；每一步把一小块 Prefill 和正在进行的若干 Decode 混在同一个批次里跑。Prefill 借机填满 Decode 留下的算力空隙，二者共生。

    收益：平滑了 Prefill 对 Decode 的冲击，TPOT 更稳定；同时让 memory-bound 的 Decode 和 compute-bound 的 Prefill 互补，整机利用率更高。代价是单个长 Prefill 的完成被切散，需要权衡块大小。

    Prefill
    Decode
    混合调度，稳住 TPOT

    7.6 PD 分离（Prefill / Decode Disaggregation）

    问题：Prefill 和 Decode 性格相反，硬塞在同一组 GPU 上总会互相干扰——要么 Prefill 抢了 Decode 的节奏，要么为了照顾延迟而牺牲吞吐。而且两者对硬件的偏好不同（Prefill 要算力，Decode 要带宽和显存）。

    做法：把两个阶段拆到不同的 GPU/节点。一组机器专做 Prefill，算完把 KV Cache 通过高速互联传给另一组专做 Decode 的机器。各自独立扩缩容、独立调优。

    收益：消除阶段间干扰，TTFT 和 TPOT 都更可控，两类资源可按需配比。代价是要跨节点传输 KV Cache（量不小，依赖高速网络），架构也更复杂。这是大规模在线服务（如超大模型 API）近年的重要方向。

    隔离两阶段，各自最优

    7.7 投机解码（Speculative Decoding）

    问题：Decode 受访存限制，每步读一遍大模型权重只产出 1 个 token，算力大量闲置。

    做法：用一个小而快的草稿模型（或其他机制）一口气猜出未来若干个 token，再让大模型一次性并行验证这串猜测——验证 K 个 token 和 Decode 1 个 token 的访存成本几乎相同（都只读一遍权重），但若猜对就一次落地多个 token。

    收益：在不改变输出分布的前提下，把"每次权重读取的产出"从 1 个 token 提到多个，等于把闲置算力换成了速度，TPOT 显著下降。收益取决于草稿模型的命中率。

    主要改善 TPOT（用闲置算力换访存效率）

    7.8 注意力结构层面：GQA / MQA

    问题：KV Cache 大小正比于 KV 头数（第 4 章），它是 Decode 访存和显存的大头。

    做法：让多个 Query 头共享同一组 K、V 头。MQA 是所有头共享一组（最省），GQA 是分组共享（在质量与开销间折中，现已成主流）。

    收益：KV Cache 直接缩小数倍 → Decode 每步访存更少、显存能放下更多并发。这是在模型设计阶段就内建的优化，几乎所有现代大模型默认采用。

    从源头缩小 KV

    第 8 章 · 实战

    真实框架如何处理两阶段

    理论落到代码，会看到一个清晰的执行循环：调度器在每一步决定"这一拍跑哪些 Prefill、哪些 Decode"，KV Cache 管理器决定"谁的缓存放哪、能不能复用"。这一章用框架视角把前面的概念串成一个可运行的心智模型——只求理解骨架，不抠实现细节。

    8.1 主流框架速览

    框架定位 / 特点与两阶段相关的招牌能力
    vLLM开源、通用、易用PagedAttention、continuous batching、prefix caching、chunked prefill
    TensorRT-LLMNVIDIA 官方，极致性能编译优化的 in-flight batching、分页 KV、量化
    SGLang面向复杂提示 / AgentRadixAttention（基于前缀树的高效 KV 复用）
    TGIHugging Face 服务化continuous batching、生产级服务封装
    名字不同，内核一致各框架术语略有差异（in-flight batching ≈ continuous batching，RadixAttention 是 prefix caching 的一种实现），但都围绕第 7 章那几件事：拼批、管 KV、复用前缀、混跑两阶段。看懂了原理，换任何框架都能快速上手。

    8.2 一个迭代式调度循环（心智模型）

    把一台推理引擎想象成一个不停转的循环，每转一圈叫一个 step（一次模型前向）。每个 step，调度器做三件事：

    1. 挑活：从等待队列里挑出能跑的请求。有新请求要做 Prefill，有在途请求要做 Decode——在显存预算内尽量塞满这一批。
    2. 组批：把选中的 Prefill 块和 Decode token 拼成一个混合批次（开启 chunked prefill 时），或分开成 Prefill 批与 Decode 批。
    3. 执行 + 回收：跑一次前向；为新 token 追加 KV；已生成完的请求退出并释放它的 KV 页，归还给显存池供后来者使用。

    8.3 一段把概念串起来的伪代码

    while server_running:
    # 1) 调度：在 KV/显存预算内挑请求组批
    batch = scheduler.pick(
        waiting_queue, running_queue,
        kv_budget = free_kv_blocks()
    )
    # batch 里可能同时含：若干 Prefill 块 + 若干 Decode token
    #   —— continuous batching + chunked prefill

    # 2) 一次模型前向（这一个 step）
    logits = model.forward(batch, kv_cache)   # 复用/追加 KV

    # 3) 采样下一个 token
    new_tokens = sample(logits)

    # 4) 写回 & 回收
    for req in batch:
        kv_cache.append(req, new_tokens[req])   # 追加新 KV 页
        if req.finished():                      # 命中结束符/长度上限
            kv_cache.free(req)                  # 释放页，归还显存池
            running_queue.remove(req)

    对照前几章：kv_budget 体现了"显存常是并发上限"（第 4 章）；混合 batch 体现了 chunked prefill 和 continuous batching（第 7 章）；kv_cache.append/free 就是分页 KV 的分配与回收（第 7 章）。整个循环，就是把 compute-bound 的 Prefill 和 memory-bound 的 Decode 编排在一起的工程答案。

    8.4 调参时该想什么

    • max batch / max KV blocks：调大→吞吐升、延迟与显存压力升。对着第 6 章的 SLO 找工作点。
    • 是否开 chunked prefill、块大小：长 prompt、对 TPOT 稳定性敏感的场景值得开。
    • 是否开 prefix caching：有固定 system prompt / 多轮对话 / Agent 模板时收益大。
    • 是否上投机解码、PD 分离：追求极致 TPOT 或超大规模在线服务时考虑，代价是复杂度和额外资源。
    从框架反推原理当你在 vLLM 日志里看到 "running: 32 reqs, pending: 5, GPU KV cache usage: 88%" 这类信息，其实就是本手册的全部概念在实时跳动：32 个请求在混合批里滚动 Decode，5 个在排队等 Prefill，KV Cache 快满了——再多就要触发抢占或排队。看懂这行日志，说明你已经把两阶段吃透了。

    第 9 章 · 总结

    总结与速查表

    一页纸把全篇收束。先用一条逻辑链把所有概念串起来，再附上随手可查的对照表。

    9.1 一条逻辑链贯穿全篇

    语言模型靠自回归生成（第 1 章）——已知的 prompt 可并行处理，未知的续写必须串行。这一条事实劈出两个阶段：

    Prefill 把整段已知输入一次算完（第 2 章），产出 KV Cache 并预测首个 token，因为算得多而并行，是 compute-bound，决定 TTFT。

    Decode 逐个 token 串行生成（第 3 章），每步只算 1 个 token 却要把权重和缓存全读一遍，是 memory-bound，决定 TPOT。

    把两者缝在一起的是 KV Cache（第 4 章）：用显存换计算，由 Prefill 写、被 Decode 读和追加，常常是并发的真正上限。

    Roofline（第 5 章）解释了为什么 batching 能救 Decode（抬高算术强度）却对 Prefill 帮助有限。指标体系（第 6 章）让我们能分别度量两阶段，并理解吞吐与延迟的根本矛盾。最后，所有系统优化（第 7 章）和真实框架（第 8 章）都在做同一件事：让 compute-bound 的 Prefill 别等、让 memory-bound 的 Decode 别浪费算力，再把两者和谐地编排在同一块 GPU 上。

    9.4 容易记混的几个点

    • 缓存的是 K 和 V，不是 Q。Q 每步都新算，没有复用价值。
    • Prefill 也是因果的，但因果掩码只限制"看谁"，不妨碍 token 并行计算；并行与否取决于 token 是否已知。
    • Decode 慢不是因为算得多，而是因为搬得多——每步只算 1 个 token，却要全量读权重和缓存。
    • 显存常比算力先成为并发瓶颈，因为 KV Cache 随并发和长度线性膨胀。
    • TTFT 看 Prefill，TPOT 看 Decode，优化前先定位是哪个指标不达标。

    9.5 一句话总纲

    同一个模型，Prefill 把已知输入并行算成 KV Cache（吃算力、定首字），Decode 借助 KV Cache 串行逐字生成（吃带宽、定吐字速度）；推理系统的全部艺术，就是用 batching、分页、缓存复用、混合调度等手段，把这两副相反的面孔安顿在同一块 GPU 上，同时满足延迟与吞吐。
