总览：Claude Code 的全景地图

先建立一张地图：Claude Code 是什么，由哪些层组成，各个概念分别"插"在系统的哪个位置。

Claude Code 是什么

Claude Code 是一个智能体编程工具（agentic coding tool）：它不是聊天框，而是一个能读你的代码库、编辑文件、执行命令、连接外部工具的"会自己干活"的程序。它运行在终端（CLI）、IDE（VS Code / JetBrains）、桌面应用和浏览器（claude.ai/code）中，内核相同。

它的本质可以一句话概括：一个大语言模型 + 一组工具 + 一个循环。模型负责推理决策，工具负责与真实世界交互（读写文件、跑命令），循环负责让"思考 → 行动 → 观察结果 → 再思考"持续进行，直到任务完成。

三层架构：核心 / 扩展 / 控制

把所有概念放进一个框架里，Claude Code 由三层组成：

① 核心层（不可替换）Agent 主循环 · 上下文窗口 · 内置工具（Read / Edit / Bash / Grep …）
② 扩展层（按需添加）CLAUDE.md（记忆）· Skills（技能）· Subagents（子代理）· Agent Teams · Hooks（钩子）· MCP（外部连接）· Plugins（打包分发）
③ 控制层（约束行为）权限规则（allow / ask / deny）· 权限模式（plan / auto …）· settings.json · 沙箱

每个概念解决什么问题

扩展层的每个机制都对应一个明确的问题，这是理解它们的最快方式：

概念解决的问题一句话定义
CLAUDE.md每次新会话 Claude 都"失忆"，重复解释项目规范每个会话自动加载的持久化指令文件
Skill反复粘贴同一套操作步骤 / 参考资料按需加载的知识与工作流，可用 /名称 触发
Subagent副任务（大量搜索、读文件）撑爆主对话上下文独立上下文中干活、只交回摘要的隔离工作者
Agent Team多条工作线需要互相讨论、自主协作多个独立 Claude Code 会话组成的协作团队
Hook"每次都必须发生"的事不能靠模型自觉生命周期事件上确定性触发的自动化
MCPClaude 看不到你的数据库 / Slack / 监控系统连接外部服务的开放协议（AI 界的 USB-C）
Plugin同一套配置要复制到多个仓库 / 分发给他人把 skills + agents + hooks + MCP 打包成可安装单元

第一部分 · 基础架构 / 第 2 章

Agent 主循环：一切的引擎

理解了主循环，其它所有概念都是它的"插件位"。本章讲：上下文窗口、系统提示、工具调用循环、以及一次完整请求的生命周期。

什么是 Agentic Loop（智能体循环）

传统聊天是"一问一答"。Claude Code 的不同之处在于：模型每次回复可以选择调用工具而不是直接回答；工具执行的结果再喂回给模型，模型据此决定下一步。如此往复，模型就能完成"读 10 个文件 → 改 3 处代码 → 跑测试 → 修复失败 → 提交"这样的长链任务。

你的指令"修复登录的 bug"
→
模型思考我需要先看代码
→
调用工具Grep("login")
→
结果返回找到 auth.ts:42
→
继续思考读文件 → 改代码 → 跑测试

↻ 循环持续，直到模型判断任务完成，输出最终回复（Stop 事件触发）

上下文窗口（Context Window）

上下文窗口是模型一次能"看见"的全部内容，是 Claude Code 中最重要的稀缺资源。一个会话的上下文由以下部分组成，按加载时机分为启动加载和运行时累积：

内容加载时机占用特点
系统提示（System Prompt）会话启动固定开销，定义 Claude Code 的行为方式
CLAUDE.md 全文 + 规则文件会话启动每次请求都在场，所以要精简（建议 < 200 行）
Skill 的名称与描述会话启动很小；技能正文只在被调用时才加载
MCP 工具名会话启动很小；完整 schema 延迟到使用时加载
对话消息 + 工具调用结果运行时累积主要消耗来源：读的每个文件、每条命令输出都进来
Hook 的输出（如有）事件触发时默认为零，仅当 hook 返回内容时注入
几乎所有架构设计都围绕"省上下文"：Skill 的渐进式披露、Subagent 的上下文隔离、MCP 的工具延迟加载、.claude/rules/ 的路径触发加载——记住这条主线，很多设计就不难理解了。

上下文满了怎么办：压缩（Compaction）

当上下文接近上限时，Claude Code 会进行压缩：把早期对话总结成摘要，释放空间。可手动 /compact 触发。压缩前后分别触发 PreCompact / PostCompact 钩子。项目根目录的 CLAUDE.md 在压缩后会自动重新注入，但仅存在于对话中的临时指令可能丢失——重要规则应写进 CLAUDE.md。

一次工具调用的完整生命周期

模型说"我要执行 npm test"之后，到命令真正运行之间，有一整条检查链：

  1. 模型发起工具调用 —— 输出结构化的调用请求（工具名 + 参数）。
  2. PreToolUse 钩子 —— 如果配置了，可以直接拦截（比如禁止 rm -rf）。
  3. 权限判定 —— 依次匹配 deny 规则 → ask 规则 → allow 规则 → 当前权限模式的默认行为（详见第 11 章）。需要时弹出确认框（触发 PermissionRequest 钩子）。
  4. 执行 —— 工具真正运行（Bash 命令可在沙箱中执行）。
  5. PostToolUse / PostToolUseFailure 钩子 —— 成功或失败后触发，可注入额外信息（如 lint 结果）。
  6. 结果回填上下文 —— 模型读到结果，进入下一轮推理。

系统提示 vs CLAUDE.md

一个常见误解：CLAUDE.md 并不在系统提示里。它是以用户消息的形式在系统提示之后注入的。这意味着它是"强烈的建议"而非"硬性约束"——模型会尽力遵守，但不保证 100%。需要硬保证的规则，用 Hook 或权限 deny 规则实现（这是贯穿全书的一条重要原则）。如确需追加系统提示级指令，可用 CLI 参数 --append-system-prompt。

模型选择

用 /model 切换模型。常用别名：opus（最强推理）、sonnet（均衡）、haiku（快而省）、opusplan（规划用 Opus、执行用 Sonnet 的混合策略）。子代理可独立指定自己的模型（见第 6 章），这是控制成本的重要手段。

第一部分 · 基础架构 / 第 3 章

内置工具：Claude 的"手和眼睛"

工具是模型与真实世界交互的唯一通道。本章按用途分类讲解内置工具，以及它们的权限特征。

工具全景

📖 读取与搜索（只读，免审批）

Read 读文件（含图片/PDF/笔记本）· Glob 按文件名模式找文件 · Grep 按内容正则搜索（基于 ripgrep）

✏️ 写入与编辑（需审批）

Write 新建/覆盖文件 · Edit 精确字符串替换 · NotebookEdit 编辑 Jupyter 单元格

⚡ 执行（需审批）

Bash 执行 shell 命令，支持后台运行与沙箱隔离；危险命令受权限规则约束

🌐 网络

WebSearch 搜索网页 · WebFetch 抓取指定 URL 内容供模型分析

🤖 委派

Agent 工具：派生子代理执行任务（v2.1.63 起由 Task 更名而来，旧名仍兼容）

📋 流程管理

TaskCreate / TaskUpdate 任务清单 · AskUserQuestion 向你提问 · EnterPlanMode / ExitPlanMode 进出规划模式 · Skill 调用技能

🧠 代码智能（LSP）

安装语言的 code intelligence 插件后，可做符号级跳转定义/找引用，并在编辑后返回类型错误诊断——比 grep 更精准省上下文

🔌 MCP 工具

所有连接的 MCP 服务器工具，命名格式 mcp__服务器__工具，与内置工具在同一循环中被调用（见第 9 章）

权限视角下的工具分类

工具的核心分界线是"是否会改变世界"：

类别工具默认行为
只读Read · Glob · Grep · 只读 Bash 命令（如 ls、git status）不需要审批，任何模式下都可自由执行
写入Write · Edit · NotebookEditdefault 模式逐次询问；acceptEdits 模式自动放行（限工作目录内）
执行Bash · PowerShell询问；可用规则按命令模式精确放行，如 Bash(npm test)
网络WebSearch · WebFetch · MCP 工具询问或按规则放行

Bash 工具的两个重要机制

后台运行

长时间运行的命令（如 dev server、watch 模式）可转入后台执行，Claude 继续干别的活，随时回头查看输出。交互式会话中可用 Ctrl+B 把正在运行的命令切到后台。

沙箱（Sandboxing）

开启沙箱后（settings 中 sandbox.enabled），Bash 命令在受限环境中运行：文件系统隔离（只能写工作目录）+ 网络隔离（仅允许放行的域名）。沙箱内的命令可以更放心地自动批准，从而实现"更自主但仍安全"的执行。这也是 auto 模式（见第 11 章）的配套设施。

工具与扩展机制的关系

理解一个关键点：Subagent、Skill、MCP 最终都表现为"工具调用"。派生子代理 = 调用 Agent 工具；模型自动使用技能 = 调用 Skill 工具；查询数据库 = 调用 mcp__postgres__query 工具。主循环只认识"工具"这一种动作，这让整个系统保持了统一的权限、钩子与审计模型——所有扩展机制都自动获得同一套控制能力。

记忆体系：CLAUDE.md、规则与自动记忆

每个会话从空白上下文开始。两套机制让知识跨会话延续：你写的 CLAUDE.md，和 Claude 自己记的 Auto Memory。

两套记忆系统对比

CLAUDE.md 文件Auto Memory 自动记忆
谁来写你（人工维护）Claude（自动积累）
内容指令与规范："永远用 pnpm"经验与发现：构建命令、调试心得
作用域项目 / 用户 / 组织按 git 仓库，同仓库的 worktree 共享
加载方式每会话全文加载每会话加载 MEMORY.md 前 200 行 / 25KB
存储位置项目内 / ~/.claude/~/.claude/projects/<项目>/memory/

CLAUDE.md 的层级与加载顺序

CLAUDE.md 可以放在四个层级，按"作用域从宽到窄"的顺序依次进入上下文（后加载的更靠近对话，影响力更直接）：

① 管理策略组织级，IT 部署
/Library/Application Support/ClaudeCode/
→
② 用户级个人偏好，所有项目
~/.claude/CLAUDE.md
→
③ 项目级团队共享，进版本库
./CLAUDE.md
→
④ 本地级个人项目偏好，gitignore
./CLAUDE.local.md

加载机制的三个细节：

  • 向上爬目录：从工作目录向上走到文件系统根，沿途所有 CLAUDE.md 在启动时全部加载（根目录的在前，离你最近的在后）。所有层级是叠加关系而非覆盖。
  • 子目录懒加载：工作目录之下子目录里的 CLAUDE.md 不在启动时加载，而是当 Claude 读取该目录下的文件时才进入上下文——monorepo 里给每个包放局部说明就靠这个。
  • @ 导入：CLAUDE.md 中可写 @README.md、@docs/git-instructions.md 导入其他文件，最多递归 4 层；导入的文件同样在启动时全文加载（只是便于组织，不省上下文）。

.claude/rules/：模块化与路径触发

当 CLAUDE.md 变大时，把指令拆成 .claude/rules/*.md 主题文件（如 testing.md、security.md，支持子目录递归发现与符号链接共享）。其杀手锏是 paths frontmatter——规则只在 Claude 操作匹配文件时才加载，按需占用上下文：

<!-- .claude/rules/api-design.md -->
---
paths:
  - "src/api/**/*.ts"
---
# API 开发规则
- 所有端点必须做输入校验
- 使用统一错误响应格式

不带 paths 的规则启动即加载，优先级等同 .claude/CLAUDE.md。用户级规则放 ~/.claude/rules/，先于项目规则加载。

Auto Memory：Claude 的自动笔记

Claude 在工作中会自己判断"哪些信息将来有用"并记下来（构建命令、调试结论、你的偏好），默认开启。结构是一个索引 + 多个主题文件：

~/.claude/projects/<项目>/memory/
├── MEMORY.md          # 索引，每会话加载前 200 行 / 25KB
├── debugging.md       # 主题文件，按需用 Read 工具读取
└── api-conventions.md

对话中说"记住：API 测试需要本地 Redis"，Claude 会写入 auto memory；说"把这条加进 CLAUDE.md"，则写入指令文件。所有记忆都是纯 markdown，可随时用 /memory 查看、编辑、删除。

实用命令与原则

命令 / 设置作用
/init分析代码库自动生成 CLAUDE.md 初稿；已有则提出改进建议；会读取已存在的 AGENTS.md、.cursorrules 等
/memory列出当前会话加载的全部记忆文件，可打开编辑、开关 auto memory
claudeMdExcludessettings 中按 glob 排除不相关的 CLAUDE.md（大 monorepo 中跳过其他团队的文件）
autoMemoryEnabled: false关闭自动记忆（或环境变量 CLAUDE_CODE_DISABLE_AUTO_MEMORY=1）
写好 CLAUDE.md 的三原则：短（单文件 < 200 行，长了拆 rules 或 skill）、具体（"用 2 空格缩进" 优于 "格式规范"）、无冲突（两条矛盾指令会让模型随机选一条）。记住：CLAUDE.md 是上下文不是配置——必须硬执行的规则用 Hook（第 8 章）或 deny 权限规则（第 11 章）。
第二部分 · 扩展机制 / 第 5 章

Skills（技能）与斜杠命令

Skill 是最灵活的扩展机制：一个 markdown 文件就能给 Claude 增加知识、流程或可触发的工作流。自定义斜杠命令现已并入 Skills 体系。

什么是 Skill

Skill 是一个包含指令的 SKILL.md 文件（可附带脚本与参考资料）。创建后有两种触发方式：你手动输入 /技能名，或 Claude 在判断相关时自动加载。什么时候该建技能？当你第三次往聊天里粘贴同一套操作步骤，或者 CLAUDE.md 里某段内容已经长成"流程"而非"事实"时。

概念合并：早期版本中"自定义斜杠命令"（.claude/commands/deploy.md）与 Skill 是两个东西，现已合并——两种写法都生成 /deploy 命令且行为一致，旧的 commands 文件继续有效。Skill 额外提供：支撑文件目录、调用控制 frontmatter、自动触发能力。Claude Code 的 Skill 遵循 Agent Skills 开放标准，可跨 AI 工具复用。

核心机制：渐进式披露（Progressive Disclosure）

这是 Skill 省上下文的关键设计，分三层加载：

第 1 层：启动时只加载 name + description
（每条上限 1536 字符）
→
第 2 层：触发时SKILL.md 正文全文
进入上下文
→
第 3 层：按需正文引用的脚本/文档
由 Claude 用工具读取执行

所以长篇参考资料放进技能几乎零成本——不用时只有一行描述在场。Claude 依靠 description 与你的任务匹配来决定是否自动加载，描述写得准，触发才可靠。

文件结构与存放位置

.claude/skills/
└── deploy/
    ├── SKILL.md          # 必需：指令正文 + frontmatter
    ├── checklist.md      # 可选：支撑文档
    └── scripts/
        └── verify.sh     # 可选：可执行脚本
位置作用域
.claude/skills/（项目）团队共享，随版本库分发
~/.claude/skills/（用户）个人所有项目可用
插件内置带命名空间，如 /my-plugin:review

同名冲突时按优先级取一个：managed > 用户 > 项目。

Frontmatter 字段详解

字段作用
name显示名称（命令名一般取自目录名）
descriptionClaude 判断"何时自动使用"的依据，建议把关键场景写在最前
argument-hint输入 /命令 时给你的参数提示
disable-model-invocation: true禁止 Claude 自动调用，只能你手动触发——有副作用的操作（部署、发消息）务必加上；同时描述不再占上下文
user-invocable: false从 / 菜单隐藏（纯背景知识类技能用）
allowed-tools技能运行期间免审批可用的工具列表
model执行该技能时切换的模型
context: fork在隔离的子代理上下文中运行该技能，结果以摘要返回（不污染主对话）
agent配合 fork 指定执行环境：Explore、Plan、general-purpose 或自定义子代理

参数替换

技能正文中可使用占位符接收命令参数：$ARGUMENTS 展开为全部参数字符串，$0、$1 取第一、第二个参数。若正文没写 $ARGUMENTS，参数会以 ARGUMENTS: <值> 形式自动附加在末尾。

<!-- .claude/skills/fix-issue/SKILL.md -->
---
name: fix-issue
description: 按编号修复 GitHub issue。当用户要求修复某个 issue 时使用
argument-hint: "[issue 编号]"
allowed-tools: Bash(gh issue view:*), Bash(gh issue list:*)
---
用 `gh issue view $ARGUMENTS` 查看 issue 详情，
理解问题后定位相关代码，修复并编写回归测试，
最后用约定格式提交：`fix: #$ARGUMENTS 描述`

使用：输入 /fix-issue 1234，或直接说"修复 issue 1234"让 Claude 自动匹配到该技能。

内置命令 vs 自定义技能

注意区分两类以 / 开头的东西：内置命令是 Claude Code 自带的控制指令，技能是你定义的提示词扩展。常用内置命令速查：

命令作用命令作用
/init生成 CLAUDE.md/compact压缩上下文
/memory查看/编辑记忆/clear清空对话
/model切换模型/rewind回退到检查点
/permissions管理权限规则/mcp管理 MCP 连接

另有官方捆绑技能开箱即用，如 /code-review（代码评审）、/debug（调试）、/batch（批量任务）。
第二部分 · 扩展机制 / 第 6 章

Subagent（子代理）：上下文隔离的工作者

子代理是在独立上下文窗口中运行的"专职 Claude"：它在自己的窗口里翻几十个文件、跑大量搜索，最后只把结论交回主对话。

为什么需要子代理

核心问题是上下文污染：让主对话直接去"调研整个代码库的认证逻辑"，会塞进几十个文件的内容，而你之后根本不会再看它们。子代理把这些"过程性消耗"隔离在自己的窗口里，主对话只收到摘要。三大收益：省上下文、可并行（多个子代理同时跑）、可约束（限定工具、模型与权限）。

运行机制

主代理遇到匹配某个子代理
description 的任务
→
调用 Agent 工具传入任务描述
（即派生 spawn）
→
子代理独立运行全新上下文 + 自己的系统提示
+ 受限工具集，跑自己的循环
→
返回摘要只有最终结果
回到主对话

子代理启动时的上下文包括：自己的系统提示（即定义文件的正文，不是完整的 Claude Code 系统提示）、skills 字段预载的技能全文、CLAUDE.md 与 git 状态（内置的 Explore/Plan 例外，二者不加载这些）、以及主代理在派生时传入的任务说明。它不继承主对话历史。

定义一个子代理

子代理是带 frontmatter 的 markdown 文件，放在 .claude/agents/（项目级）或 ~/.claude/agents/（用户级），也可用 /agents 命令交互式创建管理：

<!-- .claude/agents/code-reviewer.md -->
---
name: code-reviewer
description: 代码评审专家。写完或修改代码后主动使用（use proactively）
tools: Read, Grep, Glob, Bash
model: sonnet
---
你是资深代码评审员。检查：可读性、命名、重复代码、
错误处理、密钥泄露、测试覆盖。按 严重/警告/建议 三级输出。
description 是主代理决定"何时委派"的唯一线索。想让它被主动使用，就在描述里写明"use proactively"或"写完代码后立即使用"。

Frontmatter 字段全表

内置子代理

名称用途特点
Explore只读搜索调研不加载 CLAUDE.md，轻量快速，适合"找到 X 在哪"

使用方式与典型模式

  • 自动委派：Claude 根据任务与 description 匹配自动派生。
  • 显式指定："用 code-reviewer 子代理检查我刚才的改动"，或 @ 提及。
  • 并行扇出："分别用三个子代理调研认证、支付、通知模块的实现"——互不依赖的调研并行跑，时间减半以上。
  • 读写分离：调研用只读的 Explore（haiku 便宜跑量），实现留给主对话（opus 保质量）——这也是控制成本的常用手法。
子代理与主对话之间只有"任务下发 + 结果返回"两次通信，中途不能对话。需要多方持续讨论协作的场景，用下一章的 Agent Teams。
第二部分 · 扩展机制 / 第 7 章

Agent Teams：多会话协作实验性

子代理是"主从结构"，Agent Team 是"对等团队"：多个完全独立的 Claude Code 会话共享任务列表、互发消息、自主协调。

与 Subagent 的本质区别

维度Subagent 子代理Agent Team 代理团队
架构运行在你的会话之内每个队员是独立的 Claude Code 会话
通信只向主代理汇报结果队员之间直接互发消息
协调主代理统一调度共享任务列表 + 自主认领
成本较低（结果摘要回传）较高（每个队员都是完整实例）
适合"只要结果"的聚焦任务需要讨论、互相挑战、分工开发的复杂工作

工作机制

Lead 会话拆解任务
创建共享任务列表
→
队员 A认领任务 1
独立上下文
⇄
队员 B认领任务 2
可向 A 发消息质疑
⇄
队员 C认领任务 3
空闲时触发 TeammateIdle

三个机制支撑协作：共享任务列表（带依赖关系，队员自主认领）、点对点消息（发现冲突直接沟通）、集中视图（你从一处监控所有队员）。相关钩子：TeammateIdle（队员即将空闲）、TaskCreated / TaskCompleted。

典型场景

  • 竞争性假设调试：三个队员各持一种 bug 假设并行排查，互相用证据反驳——避免单线程"先入为主"。
  • 多视角并行评审：安全、性能、测试覆盖三个评审员同时审一个 PR。
  • 分模块开发：每人独立负责一个模块，依赖处通过消息对齐接口。
实验性功能，默认关闭。经验法则：先用子代理，当你发现"并行的子代理需要互相通信"或上下文频繁见顶时，再升级到 Agent Teams。背景代理（多个独立会话集中监控）与 git worktree 并行（第 12 章）是两种更轻的替代方案。
第二部分 · 扩展机制 / 第 8 章

Hooks（钩子）：确定性的自动化

CLAUDE.md 里写"每次编辑后跑 lint"是请求，模型可能忘；Hook 是保证——在生命周期事件上必然执行。这是"软引导"与"硬约束"的分界线。

三个组成部分

一条钩子配置由三层构成：事件（event）决定何时触发 → 匹配器（matcher）过滤触发条件 → 处理器（handler）执行实际动作。

事件如 PreToolUse
（生命周期节点）
→
匹配器如 "Bash" 或 "Edit|Write"
（过滤工具名等）
→
处理器shell 命令 / HTTP / MCP 工具
/ 模型提示 / 子代理
→
决策放行 / 拦截 / 注入信息
（通过退出码或 JSON）

生命周期事件总表

事件覆盖会话全程，常用的标粗：

阶段事件触发时机
会话SessionStart / SessionEnd会话开始（含恢复）/ 结束
Setup以 --init-only 等启动时，用于 CI 一次性准备
InstructionsLoaded / ConfigChangeCLAUDE.md / 规则加载时；配置文件变化时
输入UserPromptSubmit你提交提示词后、模型处理前（可注入或拦截）
UserPromptExpansion命令展开为提示词前（可拦截展开）
工具PreToolUse工具执行前——可拦截，最常用的安全闸门
PostToolUse / PostToolUseFailure工具成功 / 失败后（跑 lint、格式化的主场）
PostToolBatch一批并行工具调用全部完成后
PermissionRequest / PermissionDenied权限弹窗出现时 / 被 auto 模式拒绝时（可返回 retry）
子代理SubagentStart / SubagentStop子代理派生 / 结束时
TeammateIdle · TaskCreated · TaskCompletedAgent Team 队员空闲 / 任务创建 / 完成
输出Stop / StopFailureClaude 完成回复时（验收检查的好位置）/ API 错误终止
Notification / MessageDisplay发送通知时 / 显示消息文本时
环境PreCompact / PostCompact上下文压缩前 / 后
FileChanged / CwdChanged被监视文件变化 / 工作目录变化
WorktreeCreate / WorktreeRemove · Elicitation(Result)worktree 创建/移除；MCP 请求用户输入时

配置与示例

钩子配置在 settings.json 的 hooks 键下（用户级 / 项目级 / 本地级均可，多处配置全部生效、按事件合并）。经典示例——拦截危险删除命令：

// .claude/settings.json
{
  "hooks": {
    "PreToolUse": [{
      "matcher": "Bash",
      "hooks": [{ "type": "command", "command": "./block-rm.sh" }]
    }],
    "PostToolUse": [{
      "matcher": "Edit|Write",
      "hooks": [{ "type": "command", "command": "npx prettier --write \"$file\"" }]
    }]
  }
}

匹配器规则：精确字符串（"Bash"）、正则（"Edit|Write"、"mcp__.*__write.*"）、"*" 或省略表示全部匹配。匹配对象是输入 JSON 中的字段（工具事件为 tool_name）。部分事件（如 UserPromptSubmit、Stop）不支持匹配器，每次都触发。

五种处理器类型

类型机制适用
command运行 shell 命令；stdin 收到事件 JSON，用退出码和 stdout 表态最常用：lint、格式化、拦截、通知
http向 HTTP 端点 POST 事件数据对接团队中央服务、审计日志
mcp_tool调用已连接 MCP 服务器上的工具复用现有集成（如发 Slack）
prompt把问题交给一个 Claude 模型做单轮"是/否"判断规则难以正则化的智能审查
agent 实验派生子代理，可用 Read/Grep 等工具核实后再返回决策需要查证据的复杂判定

钩子如何表态：退出码与 JSON

command 钩子通过两种方式影响主循环：

  • 退出码 0 + 无输出：不表态，走正常权限流程（沉默 ≠ 批准）。
  • 退出码 2：阻断该操作，stderr 内容作为原因反馈给 Claude，让它换条路。
  • stdout 输出 JSON：精细控制——{"decision":"block","reason":"…"} 拦截并说明原因；PreToolUse 还可返回允许决策直接放行；部分事件支持注入 systemMessage 等附加上下文。
设计原则："必须每次发生、不需要思考"的事用 Hook（格式化、拦截、记录）；"需要模型理解和判断"的事用 Skill。Hook 的输出会进入上下文：PostToolUse 钩子把 lint 错误喂回去，Claude 看到就会自己修。
第二部分 · 扩展机制 / 第 9 章

MCP：连接外部世界的协议

Model Context Protocol 是一个开放标准，让 Claude 用统一方式接入数据库、Slack、GitHub、Figma 等任意外部系统——常被比作"AI 应用的 USB-C 接口"。

架构：客户端 / 服务器

Claude CodeMCP 客户端（Host）
⇄
MCP 服务器GitHub · Postgres · Slack…
把外部系统包装成标准接口
⇄
外部系统API · 数据库 · SaaS

服务器向客户端暴露三类原语（primitive）：

原语是什么例子
Tools 工具可调用的操作（最核心）create_issue、query_database
Resources 资源可读取的数据，可用 @服务器:资源 引用文件、schema、文档
Prompts 提示服务器预定义的提示模板，以斜杠命令形式出现/mcp__github__pr_review

MCP 工具进入主循环后与内置工具完全同等：统一命名为 mcp__服务器名__工具名（如 mcp__github__create_issue），同样受权限规则与钩子约束。默认开启 tool search：启动时只加载工具名，完整 schema 用到才取，闲置集成几乎不占上下文。

三种传输方式（Transport）

传输运行位置添加命令
stdio本地子进程（最常见）claude mcp add --transport stdio 名称 -- npx 服务器包名
http远程服务（流式 HTTP，推荐）claude mcp add --transport http notion https://mcp.notion.com/mcp
sse远程服务（Server-Sent Events，旧式）claude mcp add --transport sse asana https://mcp.asana.com/sse

远程服务器普遍用 OAuth 认证：运行 /mcp 在浏览器中完成授权，令牌安全存储并自动刷新。本地服务器用 --env KEY=值 传 API 密钥。断线的远程服务器会自动重连。

三个作用域（Scope）与优先级

作用域存储位置适用
local（默认）~/.claude.json 中按项目记录，仅本机本项目个人实验、含密钥的配置
project项目根 .mcp.json，进版本库团队共享团队统一的集成（首次使用需确认安全提示）
user~/.claude.json，本机所有项目可用个人通用工具

同名服务器优先级：local > project > user。添加时用 --scope project 指定作用域；claude mcp list 查看全部，/mcp 会话内管理连接状态与令牌成本。

# .mcp.json（项目作用域，团队共享）
{
  "mcpServers": {
    "github": {
      "type": "http",
      "url": "https://api.githubcopilot.com/mcp/"
    },
    "postgres": {
      "type": "stdio",
      "command": "npx",
      "args": ["-y", "@bytebase/dbhub", "--dsn", "${DB_URL}"]
    }
  }
}

MCP 能做什么：一个串联示例

连接 Jira、Sentry、Postgres、Slack 后，一句话即可触发跨系统工作流："实现 Jira 单 ENG-4521 描述的功能 → 查 Sentry 确认相关报错 → 从数据库找 10 个受影响用户 → 在 GitHub 开 PR → 给团队 Slack 发摘要"。每一步都是一次 MCP 工具调用，由主循环自动串联。

MCP 服务器能读你的数据、以你的身份操作外部系统。只连可信来源，警惕"提示注入"风险——外部内容（如网页、issue 正文）可能包含诱导指令。敏感操作配合 ask 权限规则二次确认。
MCP 与 Skill 是黄金搭档（见第 14 章）：MCP 提供"连接"，Skill 提供"用法知识"——比如 MCP 连上数据库，再用一个 Skill 教 Claude 你的表结构和查询惯例。另外 Claude Code 自身也可作为 MCP 服务器供其他应用调用（claude mcp serve）。

Plugins 与 Marketplace：打包与分发

前面各章的扩展机制都是"散件"。Plugin 把它们装进一个盒子，一条命令安装；Marketplace 是分发这些盒子的货架。

Plugin 能打包什么

Skills 技能

带命名空间的命令与知识，如 /my-plugin:deploy，避免与其他插件冲突

Agents 子代理

预配置的专职代理（注：插件中的 agent 不支持 hooks / mcpServers / permissionMode 字段，需要时复制出来自行维护）

Hooks 钩子

随插件安装即生效的自动化规则

MCP 服务器

插件启用时自动连接的外部集成

插件目录结构

my-plugin/
├── .claude-plugin/
│   └── plugin.json        # 元数据清单：name、description、version、author
├── skills/
│   └── deploy/SKILL.md    # → /my-plugin:deploy
├── agents/
│   └── reviewer.md
├── hooks/
│   └── hooks.json
└── .mcp.json              # 随插件连接的 MCP 服务器

Marketplace：分发机制

市场本质是一个包含 .claude-plugin/marketplace.json 清单的 git 仓库（或 URL），列出可安装的插件。任何人都可以建市场——公司内部市场、社区市场皆可：

# 添加市场（git 仓库即市场）
/plugin marketplace add 组织名/插件市场仓库

# 浏览与安装
/plugin install 插件名@市场名

安装后可在 /plugin 菜单中启用/禁用。团队协作时，可在项目 settings 中声明依赖的市场与插件，成员打开项目即自动提示安装——实现"新人零配置上手"。

何时用 Plugin

  • 第二个仓库需要同一套配置时——这是官方给出的明确触发信号。
  • 团队标准化：统一的评审流程、部署技能、安全钩子，一次安装全员一致。
  • 能力分发：把你打磨好的工作流分享给社区。
理解定位：Plugin 本身不提供新能力，它是打包层——内容物仍是 skills / agents / hooks / MCP。先把散件调通，再考虑打包。
第三部分 · 控制与工作流 / 第 11 章

权限与设置：安全与自主的平衡

权限系统回答一个问题：哪些操作可以直接做，哪些要先问你？答案由"模式"定基线、"规则"做微调、"设置文件"分层管理。

settings.json 的层级

设置文件从高到低分五层，高层覆盖低层（managed 不可被下层推翻）：

层级位置用途
① 管理策略 managed系统级 managed-settings.json（IT 部署，macOS 在 /Library/Application Support/ClaudeCode/）企业强制规则，个人无法覆盖
② 命令行参数--permission-mode 等 CLI flag单次会话覆盖
③ 本地项目.claude/settings.local.json（gitignore）个人在本项目的私有配置
④ 共享项目.claude/settings.json（进版本库）团队共享配置
⑤ 用户级~/.claude/settings.json个人全局默认

权限规则：allow / ask / deny

规则语法为 工具名(匹配模式)，三种规则按 deny → ask → allow 的顺序判定，deny 一票否决（任何模式下都生效）：

// .claude/settings.json
{
  "permissions": {
    "allow": [
      "Bash(npm run test:*)",      // 放行所有 test 子命令
      "Read(~/.zshrc)"
    ],
    "ask": [ "Bash(git push:*)" ],  // 推送必须人工确认
    "deny": [
      "Read(./.env)",               // 密钥文件禁止读取
      "Bash(curl:*)",
      "WebFetch"
    ],
    "additionalDirectories": [ "../docs/" ],
    "defaultMode": "acceptEdits"
  }
}

会话内用 /permissions 查看与编辑规则；批准弹窗时选"始终允许"也会写入规则。

六种权限模式

模式定基线，Shift+Tab 在 CLI 中循环切换 default → acceptEdits → plan：

模式免审批范围适用
default仅只读操作入门、敏感工作
acceptEdits读取 + 文件编辑 + 常见文件系统命令（mkdir/mv/cp…，限工作目录内）边写边在编辑器里看 diff 的迭代
plan仅只读（产出计划，不动代码）动手前的调研与方案设计
auto 研究预览几乎全部，由后台分类器审查每个动作长任务、减少打断（详见下文）
dontAsk仅预批准规则内的操作，其余直接拒绝CI、锁死环境的脚本
bypassPermissions全部（跳过所有检查）仅限隔离容器/VM，root 下拒绝启动

auto 模式：分类器代替弹窗

auto 模式用一个独立的分类器模型在后台审查每个动作，替代人工弹窗。它默认信任你的工作目录与 git remote，拦截越界行为：curl | bash、向外部端点发送敏感数据、生产部署、强推 main、删除会话前已存在的文件等。你在对话中说的边界（"先别 push"）也会被当作拦截信号。连续拦 3 次或累计 20 次会回退到人工弹窗模式。

受保护路径

无论什么模式（bypass 除外），写入 .git、.claude、.vscode、.bashrc、.mcp.json 等关键路径永不自动批准——防止仓库状态和 Claude 自身配置被意外破坏，allow 规则也无法预批准这些路径。

沙箱：第四道防线

权限系统之外，可开启 OS 级沙箱让 Bash 命令在文件系统与网络双重隔离中运行。四道防线叠加：权限规则（哪些操作可做）→ 权限模式（询问频率基线）→ Hooks（自定义拦截逻辑）→ 沙箱（OS 级硬隔离）。

第三部分 · 控制与工作流 / 第 12 章

Workflow：高效工作流模式

"Workflow"不是一个功能开关，而是把前面所有机制组合起来的使用模式。本章是实践层：怎么把活干好、干快、干safe。

规范流程：探索 → 计划 → 编码 → 提交

对非平凡任务，最重要的工作流是不让 Claude"上来就写代码"：

① 探索读相关文件、理解现状
明确说"先不要写代码"
→
② 计划Plan Mode 下产出方案
你审阅、修改、批准
→
③ 编码按计划实现
边做边自我验证
→
④ 提交跑测试、写 commit
开 PR、更新文档

Plan Mode 的角色

Plan Mode（Shift+Tab 切换，或单条消息前缀 /plan，或 claude --permission-mode plan 启动）是这个流程的制度保障：Claude 只能读和分析，不能改任何文件。计划就绪后它会呈现方案，你可以选择：批准并进入 auto / acceptEdits / 逐项审批模式开始执行，或继续给反馈打磨计划，还可按 Ctrl+G 在编辑器中直接修改计划文本。事前花在计划上的一分钟，胜过事后返工的十分钟——方向错误在计划阶段纠正成本最低。

测试驱动（TDD）工作流

智能体编程中威力最大的模式之一——给 Claude 一个可自动验证的目标：

  1. 让 Claude 先根据预期输入输出写测试（明说"现在是 TDD，不要写实现"）。
  2. 运行测试确认全部失败，提交测试。
  3. 让 Claude 写实现使测试通过，不许改测试。它会自动进入"写码 → 跑测试 → 修复"的循环，直到全绿。

同理可用于其他可验证目标：视觉稿（截图对比）、lint 零报错、类型检查通过。有明确判据，循环就能自我收敛。

上下文管理工作流

手段时机
/clear切换到无关新任务时，清空重来（最便宜的选择）
/compact任务连续但上下文吃紧时压缩；可附指示如 /compact 保留代码改动细节
/context随时查看各部分占用，定位"吃上下文的大户"
子代理扇出大调研拆给 Subagent，主对话只收结论（第 6 章）
/rewind 检查点每次编辑前自动存档；走偏时回退代码 / 对话到任意检查点，比 git 更细粒度
--continue / --resume恢复上次 / 任选历史会话，跨天接续工作

并行工作流：git worktree

多个 Claude 会话同时改一个目录会互相踩踏。git worktree 给每个会话一份独立的工作副本（共享 git 历史）：

git worktree add ../proj-feature-a feature-a
cd ../proj-feature-a && claude   # 会话 A 做功能
# 另一个终端：会话 B 同时修 bug，互不干扰

子代理的 isolation: "worktree"、Agent Teams 也是建立在同一思想上。桌面端则内置了"并行会话 + Git 隔离"的图形化支持。

自动化工作流：headless 模式

加 -p 即可把 Claude Code 当命令行工具嵌进脚本与 CI（详见第 13 章）：

# 单次执行，输出到 stdout
claude -p "解释这个函数做了什么" < func.ts

# 管道接龙 + 结构化输出
tail -100 app.log | claude -p "找出异常原因" --output-format json

# CI 中带预批准工具的自动修复
claude -p "修复所有 lint 错误" --allowedTools "Edit,Bash(npm run lint:*)"

日常小习惯

  • Esc 随时打断并纠偏；双击 Esc 回溯历史重新编辑提示词。
  • 具体指令一次到位："修复 #issue，先写失败测试" 优于 "看看这个 bug"。
  • 让 Claude 自己验证：跑测试、截图、diff——闭环比盲写质量高一个量级。
  • 重复的纠正 → 写进 CLAUDE.md；重复的流程 → 做成 Skill；必须发生的 → Hook。让系统随使用进化。
第三部分 · 控制与工作流 / 第 13 章

Agent SDK 与运行形态

Claude Code 的内核可以被编程调用（Agent SDK），也可以运行在终端之外的多种形态中。本章把"它还能在哪跑、怎么被集成"讲清楚。

Claude Agent SDK

Agent SDK（前称 Claude Code SDK，已更名）把驱动 Claude Code 的同一套引擎——主循环、工具、权限、MCP 支持——开放为 TypeScript / Python 库，用于构建你自己的智能体应用：

# Python
from claude_agent_sdk import query

async for message in query(prompt="分析这个仓库的架构"):
    print(message)
// TypeScript
import { query } from "@anthropic-ai/claude-agent-sdk";

for await (const msg of query({ prompt: "修复所有类型错误" })) {
  console.log(msg);
}

三个层次理解它与 CLI 的关系：

交互式 CLI人在环中
日常开发
→
headless（claude -p）无界面单次执行
脚本与 CI 的胶水
→
Agent SDK编程级控制：自定义工具、
权限回调、会话管理、子代理编排

headless 模式的关键参数：--output-format json|stream-json（结构化输出供程序解析）、--allowedTools（预批准工具）、--permission-mode、--max-turns（轮数上限）、--append-system-prompt。

运行形态全景

形态说明
终端 CLI本体。npm install -g @anthropic-ai/claude-code 或原生安装
VS Code / JetBrainsIDE 扩展：内联 diff、@ 引用文件、计划审阅、快捷键
桌面应用图形界面：并行会话 + Git 隔离、可视化 diff、应用预览、PR 监控
Web（claude.ai/code）云端沙箱中异步跑任务；Remote Control 可从手机接管本地会话
GitHub Actions在 issue / PR 里 @claude 指派任务，CI 中自动评审、修复
Slack在 Slack 中直接委派编码任务

配套机制

  • Channels：让 MCP 服务器把外部事件（CI 结果、告警、聊天消息）推送进正在运行的会话，Claude 可在你离开时自行响应——从"拉取"走向"事件驱动"。
  • 计划任务：会话内 /loop 与 cron 工具可定时/轮询执行提示词；Web 端支持云端定时任务。
  • 企业部署：可经 Amazon Bedrock / Google Vertex AI / Microsoft Foundry 接入，配合 managed settings、OTel 监控做组织级管控。
  • 状态栏与键位：/statusline 自定义底栏（上下文用量、成本、git 状态）；keybindings 文件自定义快捷键。
心智模型：一个内核，多种外壳。CLAUDE.md、Skills、MCP、权限等概念在所有形态中通用——学一次，处处适用。

选型决策与全景回顾

触发信号 → 该加什么

官方建议的渐进式构建路径——不要一开始配齐所有东西，等信号出现再加：

Claude 第二次搞错同一个约定或命令CLAUDE.md 加一条

易混概念三组对照

对照关键区别判断口诀
CLAUDE.md vs Skill前者每会话必加载，后者按需加载"永远要知道的"进 CLAUDE.md；"有时才需要的"做 Skill
Skill vs SubagentSkill 是可复用的内容，Subagent 是隔离的执行者要复用知识 → Skill；要隔离上下文 → Subagent（二者可组合：context: fork / skills: 预载）
Hook vs SkillHook 确定性触发不经思考，Skill 由模型理解执行"必须发生"用 Hook；"需要判断"用 Skill。守门规则一律 Hook

同名/同类配置的叠加与覆盖

机制多层共存时
CLAUDE.md叠加：所有层级同时进入上下文，冲突时模型自行权衡（越具体越优先）
Skills / Subagents按名覆盖：skill 为 managed > 用户 > 项目；subagent 为 managed > CLI > 项目 > 用户 > 插件
MCP 服务器按名覆盖：local > project > user
Hooks合并：所有来源的钩子都触发
Settings分层覆盖：managed > CLI > 本地项目 > 共享项目 > 用户

给上手者的建议路径

  1. 第 1 周：裸用 + /init 生成 CLAUDE.md，养成 Plan Mode 先行的习惯。
  2. 第 2–3 周：把重复纠正沉淀进 CLAUDE.md；第一个反复粘贴的流程做成 Skill；连 1–2 个最常用的 MCP（如 GitHub）。
  3. 第 2 个月：加 PostToolUse lint 钩子和 PreToolUse 安全闸；大调研习惯性交给 Explore 子代理；试 acceptEdits / auto 模式提速。
  4. 之后：worktree 并行、headless 进 CI、好配置打包成 Plugin 分发团队。
